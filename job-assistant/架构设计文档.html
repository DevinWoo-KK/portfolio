2. 三平台差异处理

BOSS直聘、猎聘、智联招聘在登录机制、页面结构、薪资编码、投递流程上存在显著差异。本系统通过 统一过滤引擎 + 平台适配脚本 两层架构来处理——80% 的过滤和评分逻辑共享，20% 的平台差异在独立脚本中处理。

2.2 BOSS PUA 薪资解码

BOSS直聘使用自定义字体将薪资数字编码为 Unicode Private Use Area (U+E031–U+E03A) 字符，直接读取 DOM 会得到乱码。解码原理：PUA 码点线性映射回 ASCII 数字。

// BOSS 薪资 PUA 解码算法 — 范围检查而非字典映射，更健壮 function decodePUA(s) { if (!s) return ''; let r = ''; for (let c of s) { let code = c.charCodeAt(0); // U+E031 → '1', U+E032 → '2', ... U+E03A → ':' if (code >= 0xe031 && code <= 0xe03a) r += String.fromCharCode(0x30 + (code - 0xe031)); else r += c; // 保留 K、·、-、元、万、薪 等非 PUA 字符 } return r; } // "K-K" → "15K-25K"

与常见的字典映射方案不同，本算法利用 PUA 码点恰好按序排列的特性做线性偏移，不遗漏字符、不需要维护映射表。

2.3 Cookie 注入

三平台均采用 Firefox cookies.sqlite → Playwright context 注入模式，避免密码登录触发验证码：

# 核心流程 1. 读取 Firefox Profile: /mnt/c/Users/Devin/.../Profiles/0owekiqg.default-release/cookies.sqlite 2. sqlite3 提取目标域名 Cookie: BOSS: .zhipin.com 猎聘: .liepin.com 智联: .zhaopin.com, passport.zhaopin.com 3. Playwright browser_context.add_cookies(cookies) 4. 验证: 导航搜索页 → 检查用户名/头像元素是否存在
设计决策：为什么不用 Camoufox？
Camoufox 依赖的 camouflage 模块在当前 WSL 环境未安装且安装复杂。Playwright Firefox + Cookie 注入同样能绕过反爬检测——使用用户真实浏览器 Profile，User-Agent、Cookie、LocalStorage 均为真实数据，维护成本更低。

2.4 投递流程陷阱

多步投递流程中，漏掉最后一步确认弹窗是最常见的静默失败——页面刷新后无任何错误提示，但投递实际未完成：

平台步骤关键陷阱
猎聘 Step 1 — 点击"聊一聊" 3 种"聊一聊"按钮，仅 2 种可见。必须加 :not(.ant-btn) 排除隐藏的 Ant Design 按钮
Step 2 — 点击"发简历"需等待弹窗渲染完成
Step 3 — 确认弹窗此步缺失 → 假投递，无声失败
智联 Step 1 — 点击"立即投递" React 渲染，evaluate.click() 无效，必须用 get_by_text().click()
Step 2 — 确认弹窗此步缺失 → 假投递，无声失败

2.5 反爬策略

错峰执行

三平台串行执行，15 分钟间隔：BOSS 09:00 → 猎聘 10:00 → 智联 11:00。避免并发触发限流。

随机间隔

页面操作间插入 3–8 秒随机延迟；JD 详情页抓取间隔 3–5 秒。避免固定节奏被识别为脚本。

验证码检测

每次关键操作后通过 browser_console 主动检测 CAPTCHA 关键词。触发时暂停 30 秒重试一次，仍失败则 Telegram 告警。

真实浏览器指纹

使用用户实际使用的 Firefox Profile，User-Agent、Cookie、LocalStorage 均为真实数据，对抗 headless 检测。

3. 评分算法设计

3.1 六层规则过滤

在 LLM 评分之前，用零成本的规则引擎过滤明确不匹配的岗位，减少 LLM Token 消耗：

2经验5 年以上 / 5–10 年 / 10 年以上应届、不限、1–3 年、3–5 年保留

薪资单位陷阱（已修复）
猎聘和智联的薪资可能是"万/年"、"K/月"、"元/月"混合格式。早期使用 max(数字列表) 做比较导致跨单位误杀——180,000 元/年 (15K/月) 会被 80K 上限错误过滤。修复方案：统一换算到「K/月」后再比较下限。

4.2 Prompt Cache 策略

这是成本优化的核心手段。DeepSeek 对 Prompt Cache 命中部分仅收 ¥0.10/百万 token（原价的 1/10–1/20）。

方法
将求职者简历摘要固化在每次 LLM 评分的 Prompt 最前面。三次评分 (BOSS / 猎聘 / 智联) 共享同一份简历前缀 → 简历段每轮 Cache Hit，仅岗位 JD 部分计费。

计算
简历 2K tokens + 100 岗 × 300 tokens JD = 32K tokens 总量
无 Cache: 32K × ¥1.5/百万 (均价) = ¥0.048
有 Cache: 简历 ¥0.0002 (Cache 价) + 30K JD × ¥1.5/百万 = ¥0.0452
简历部分省 99%，总体几乎无额外开销。

4.4 其他省钱手段

免费规则前置

6 层规则过滤全部在 Python 中完成（零 API 调用），过滤掉 40–60% 的岗位后才送 LLM。猎聘日均抓取 ~100 条，过滤后仅 ~60 条进入评分。

批处理合并

所有岗位收集完 JD 后合并为一次 LLM 调用，而非逐条调用。100 个岗位逐条调用 = 100 次 API 往返 + 100 次简历计费；合批后仅 1 次。

no_agent Cron

Cookie 健康检查、日志清理等纯脚本任务使用 no_agent=true 模式，完全不消耗 LLM Token。

仅工作日运行

所有求职 Cron 任务仅在周一至周五执行，周末大多数公司不发布新岗位。月省约 8 天费用。

5. 风险与边界

5.1 已知风险与缓解

风险影响缓解措施
BOSS 账号降权 打招呼频率过高被限制 同公司间隔 ≥5 岗；日上限 150 岗；仅投 ≥7 分
Cookie 过期 扫描 / 投递全部失败 每 2h 健康检查；08:00 预检；Telegram 即时告警
验证码升级 猎聘新增验证码类型 主动检测 CAPTCHA 关键词；失败后 Telegram 通知手动介入
DeepSeek 优惠到期 Pro 模型费用 ×4 Flash 已覆盖 90% 评分任务；Pro 仅留复杂推理
DOM 结构变更 选择器失效 独立脚本隔离影响；browser_console 注入 JS 提取更健壮

5.2 设计边界

  • 本系统是辅助决策工具，不替代求职者的最终判断。
  • LLM 评分存在主观性，5–7 分段可能有偏差，已通过双维度评分降低误判。
  • 不自动完成"线下面试预约""薪资谈判"等需要人工判断的环节。
  • 不处理需要企业 ERP / ATS 系统对接的流程。
  • 不适用于海外招聘平台 (LinkedIn / Indeed)，当前仅覆盖国内三大平台。
